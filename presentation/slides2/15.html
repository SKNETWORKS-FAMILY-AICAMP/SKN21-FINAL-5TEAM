Generation to Repair

15

버튼을 누른 뒤에는 generation,
runtime, repair가 순서대로 돈다

이 슬라이드 한 장으로 generation 파이프라인, 운영 시 retrieval 흐름, 실패 시 repair 전략까지 함께 설명할 수 있습니다.

01

Analysis

host 코드에서 auth, order API, mount 지점, retrieval source를 찾고 verified contract 후보를 수집합니다.

02

Planning

이 사이트를 어떤 계약으로 연결할지 integration plan을 만듭니다. 핵심 seam이 비면 여기서 중단합니다.

03

Compile

수정안을 하나로 섞지 않고 host-edit program과 chatbot-edit program으로 분리합니다.

04

Apply

원본 repo가 아니라 isolated runtime workspace에 먼저 적용합니다.

05

Preflight

chatbot runtime import와 wiring 문제를 validation 전에 앞당겨 조기 검출합니다.

06

Export

runtime 결과를 replay 가능한 host patch와 chatbot patch로 내보냅니다.

07

Index / Validate

Qdrant corpus 인덱싱, widget/auth/order/conversation 검증 후 artifact를 남깁니다.

핵심: LLM이 바로 소스를 덮어쓰는 게 아니라, 계약 확인 → 격리 실행 → replay/export → validation → rewind 순으로 통제합니다.

Runtime / Qdrant

운영 시점에는 widget에서 Qdrant와 host API까지 이어집니다

Widget

사용자가 질문하거나 이미지를 업로드합니다.

Chatbot API

`/api/chat`과 shared widget runtime이 요청을 받습니다.

LangGraph

order, policy, discovery subagent로 라우팅합니다.

Host / Qdrant

주문 조회는 host API, 정책·이미지 검색은 Qdrant를 조회합니다.

Image Search

우선 CLIP + Qdrant, 실패 시 Vision → text retrieval로 fallback 합니다.

Repair

실패하면 처음부터가 아니라
필요한 stage까지만 rewind

Failure Bundle

실패 stage와 artifact, 관련 파일을 묶습니다.

Diagnosis

원인과 rewind 후보를 정리합니다.

Effective Rewind

LLM 제안이 있어도 최종 rewind는 엔진이 결정합니다.

Rerun

해당 stage부터만 다시 실행해 시간을 줄입니다.
