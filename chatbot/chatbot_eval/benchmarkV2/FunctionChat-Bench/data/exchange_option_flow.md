# 교환 및 옵션변경 실행 흐름 및 시나리오 가이드

이 문서는 챗봇의 `exchange_subagent` 내에서 교환(exchange)과 옵션변경(change_option)이 어떻게 작동하는지 상세히 정의합니다.

---

## 1. 전체 실행 순서 (Workflow)

1. **의도 파악 (`order_intent_router`)**
   - 사용자의 발화에서 "교환", "사이즈 변경", "색상 바꿔줘" 등의 의도를 감지하여 `exchange_subagent`로 노드를 전환합니다.

2. **주문 식별 및 선택**
   - 대화 맥락에 `order_id`가 없으면 `get_user_orders` 도구를 호출하여 최근 주문 목록을 UI(`show_order_list`)로 표출하고 사용자의 선택을 기다립니다.

3. **도구 결정 (`_select_exchange_tool`)**
   - 입력된 키워드와 해당 주문의 현재 상태(결제완료/배송중/배송완료 등)를 분석하여 `change_option` 또는 `exchange` 중 최적의 도구를 내부적으로 선택합니다.

4. **프로세스 완료**
   - 각 도구의 시나리오에 따라 사용자 확인(HITL) 또는 추가 정보 입력 후 작업을 마무리합니다.

---

## 2. 주요 시나리오 및 분기 기준

| 구분 | **옵션변경 (change_option)** | **교환 (exchange)** |
| :--- | :--- | :--- |
| **핵심 시나리오** | **배송 전(PAID/PREPARING)** 상태에서 단순 옵션을 바꿈 | **배송 후(DELIVERED)** 제품을 받은 후 새 제품으로 교체함 |
| **분기 키워드** | "사이즈 변경", "색상 바꿔줘", "옵션 잘못 선택" | "교환해주세요", "다른 걸로 바꿔주세요", "맞교환" |
| **대상 주문 상태** | 결제완료, 상품준비중 | 배송완료 |
| **사용자 인터랙션** | 변경 가능한 옵션 목록 선택 (`new_option`) | 교환 실행 여부 최종 확인 (`confirmation`) |

---

## 3. 도구별 세부 동작 상세

### 3-1. [시나리오 A] 배송 전 옵션 변경 (`change_option`)
- **과정**:
  1. 먼저 현재 주문 내 상품에 대해 **변경 가능한 다른 옵션(색상, 사이즈 등)**이 있는지 확인합니다.
  2. 조회된 옵션 데이터를 바탕으로 사용자에게 선택 UI(`show_option_list`)를 제공합니다.
  3. 사용자가 새 옵션을 선택하면 실제 DB의 주문 정보를 즉시 업데이트합니다.
- **특징**: 배송 시작 전이므로 별도의 물품 회수 절차가 필요 없는 가벼운 프로세스입니다.

### 3-2. [시나리오 B] 배송 후 일반 교환 (`exchange`)
- **과정**:
  1. 상품이 이미 고객에게 전달된 상태이므로 **기존 물품의 회수(반품)와 새 물품의 발송(재배송)** 절차를 수반합니다.
  2. 시스템은 교환 접수 전, 사용자에게 최종 의사를 묻는 확인 UI(`confirm_order_action`)를 띄웁니다.
  3. 사용자가 승인(예, "응", "진행해")하면 상태를 업데이트하고 교환 프로세스를 시작합니다.
- **특징**: 비용 발생(배송비 등) 및 물리적 이동이 필요한 "무거운" 프로세스이므로 사용자 확인(Human-In-The-Loop)이 필수적입니다.
